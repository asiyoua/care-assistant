# CARE 助理 — 关心你的每一个想法

基于飞书多维表格的个人效率助理。说一句话就记录，AI 自动分类，定时整理，待办变飞书任务。

**CARE = Capture 随手记 + Auto-sort 自动分 + Review 定期理 + Execute 照着做**

## 前置：读取配置

每次执行前，先读取配置文件：

```
~/.care-assistant/config.json
```

如果文件不存在或缺少 `base_token` / `table_id`，则进入 Setup 流程（参见 `references/setup.md`）。

配置文件中包含：
- `base_token`：飞书多维表格 token
- `table_id`：记录表 ID
- `article_table_id`：文章收藏表 ID
- `view_id`：默认视图 ID
- `user_open_id`：当前用户 open_id

读取配置后，用配置值替换下面所有命令中的 `<base_token>`、`<table_id>`、`<article_table_id>`、`<user_open_id>`。

**网络注意：** lark-cli 连飞书 API 需要代理。命令执行前加 `https_proxy=http://127.0.0.1:7897 http_proxy=http://127.0.0.1:7897` 前缀。如果直连可达则不需要。

## 模式判断

根据用户输入判断进入哪个模式：

| 用户意图 | 模式 | CARE 环节 |
|---------|------|-----------|
| 说了要记的事 / 待办 / 灵感 / 链接 | 记录模式 | **C**apture |
| "整理一下" / "开始整理" / "处理记录" | 整理模式 | **R**eview |
| 给了转写文档 / "处理录音" | 录音模式 | **C**apture |
| 查看待办 / 查看记录 | 查看模式 | **R**eview |

## C — Capture 随手记（记录模式）

### 1. 识别内容类型

根据用户输入判断类型（有且仅有 3 种）：

| 类型 | 判断依据 | 示例 | 目标表 |
|------|---------|------|--------|
| 记录 | 待办、灵感、或其他值得记录的内容 | "明天10点开会"、"学Rust" | 记录表 |
| 文章 | 包含链接 | "https://..." | 文章收藏表 |

**重要：** 文章/链接类内容写入单独的"文章收藏"表，不混进记录表。

### 2. 提取信息

从用户输入中提取字段：

- **title**（必填）：内容摘要，超50字时需整理出标题
- **content**（可选）：完整内容，输入较长时填写
- **tags**（必填）：标签数组，从以下选项中选取：待办、灵感、其他、AI、社群、内容创作、备孕、线下。至少选一个类型标签（待办/灵感/其他），可叠加主题标签
- **start_time**（可选）：待办开始时间（仅"待办"标签时需要）
- **due_time**（可选）：待办截止时间（仅"待办"标签时需要）

时间推断规则：
- 提及日期但无具体时间：开始 = 当天 0:00，截止 = 当天 23:59
- "明天10点开会"：开始 = 明天 10:00，截止 = 明天 11:00（默认1小时）
- "明后天完成"：开始 = 明天 0:00，截止 = 后天 23:59
- 无法判断时间时，**向用户询问**，不要猜

### 3. 去重检查（写入前必做）

**每次写入前**，先读取已有记录，对比是否语义重复：

```bash
lark-cli base +record-list \
  --base-token <base_token> \
  --table-id <table_id> \
  --limit 200
```

去重规则：
- 对比新内容与已有记录的"记录标题"和"详细内容"
- 如果核心含义相同（只是措辞不同），视为重复，**跳过不写入**
- 时间不同但内容相同（如两个"上午9点开会"），也视为重复

### 4. 写入多维表格

**记录类** → 写入记录表（`table_id`）：

```bash
lark-cli base +record-upsert \
  --base-token <base_token> \
  --table-id <table_id> \
  --json '{
    "记录标题": "<title>",
    "来源": "终端",
    "处理状态": "原始",
    "标签": ["<标签1>", "<标签2>"],
    "详细内容": "<content 或空字符串>",
    "待办开始时间": "<时间 或空字符串>",
    "待办截止时间": "<时间 或空字符串>",
    "完成状态": "未完成",
    "来源文件": ""
  }'
```

**文章/链接类** → 写入文章收藏表（`article_table_id`）：

```bash
lark-cli base +record-upsert \
  --base-token <base_token> \
  --table-id <article_table_id> \
  --json '{
    "标题": "<文章标题或链接描述>",
    "链接": "<url>",
    "备注": "<补充说明 或空字符串>",
    "标签": ["<标签>"],
    "来源": "终端",
    "收藏时间": "<YYYY-MM-DD HH:mm>"
  }'
```

注意：
- 时间格式：`YYYY-MM-DD HH:mm`
- 空值传空字符串 `""`
- 标签字段是数组，可传多个标签
- 所有字段都必须填写

### 5. 待办类 → Execute（创建飞书任务）

标签含"待办"时，写入成功后：

```bash
lark-cli task +create \
  --summary "<记录标题>" \
  --due "<YYYY-MM-DD>" \
  --assignee "<user_open_id>"
```

### 6. 反馈

```
已记录 [标签]：标题内容
```

文章类额外显示收藏链接。

待办类额外告知时间安排和飞书任务链接。

## R — Review 定期理（整理模式）

### 1. 读取未处理记录

```bash
lark-cli base +record-list \
  --base-token <base_token> \
  --table-id <table_id> \
  --limit 200
```

从返回结果中筛选 `处理状态 = 原始` 的记录。如果没有，告知用户"没有需要整理的记录"。

### 2. 逐条整理

对每条"原始"记录：

1. **检查标签**：已有标签则保留，为空则根据内容自动打标签
2. **丰富内容**：标题或内容不完整时补充
3. **待办处理**：标签含"待办"且有时间 → 创建飞书任务

### 3. 更新状态

每整理完一条，更新为"已整理"：

```bash
lark-cli base +record-upsert \
  --base-token <base_token> \
  --table-id <table_id> \
  --record-id <record_id> \
  --json '{"处理状态":"已整理"}'
```

### 4. 汇报

```
整理完成，共处理 N 条记录：
- 待办 X 条（已创建飞书任务）
- 灵感 Y 条
- 其他 W 条
```

## C — Capture 录音转写（录音模式）

### 触发条件

用户说"处理录音"、"这是今天的录音"、"分析转写"，或直接给了转写文档。

### 处理流程（三层输出）

录音内容比单条记录丰富，需要分层处理：

**第一层：结构化记录 → 多维表格**

1. **读取内容**：读取用户提供的文件或文本
2. **读取已有记录**：调用 `+record-list --limit 200` 获取当前所有记录，用于去重
3. **提取可记录项**：
   - 待办事项（有行动要求的内容）
   - 灵感/想法（点子、计划、构想）
   - 值得记录的信息（知识、经验、决策）
4. **提取后去重**：
   - 提取出的记录之间互相对比，含义相同的合并为一条
   - 再与已有记录对比，语义重复的跳过
5. **逐条写入**：按记录模式的格式写入，`来源` 改为 `录音`，`来源文件` 填写文件名，使用标签系统打标签
6. **待办额外处理**：标签含"待办"的创建飞书任务

**第二层：飞书文档（提炼 + 原文，同一个文档）**

使用 `lark-cli docs +create` 创建飞书文档，标题格式："录音：{来源文件名}"

文档结构（按顺序）：

```
## 精炼提炼

### 亮点摘录
- 提炼出对话中的好段落/金句，每条用 callout 高亮块展示

### 写作话题
- 从对话中提炼可展开成文章的话题
- 每个话题含：切入点、可参考段落、适合发布的平台

### 行动要点
- 用表格汇总所有行动项和优先级

---

## 原文转写

（完整转写内容，保持原样）
```

**关键原则**：提炼在前、原文在后。提炼部分是日常看的，原文是存档备查的。飞书文档有目录导航，可以快速跳转。

**第三层：汇报**

```
录音处理完成：
- 表格记录 N 条（待办 X / 灵感 Y / 其他 Z）
- 飞书文档已创建：{doc_url}
```

### 提取原则

- 口语化表达需提炼为简洁的标题和内容
- 同一话题的多句话合并为一条
- 明确的行动项 → 待办，模糊的想法 → 灵感
- 好段落/金句 → 摘录到飞书文档（不放表格）
- 写作话题 → 摘录到飞书文档，附上切入点和适合平台
- 闲聊、重复、语气词不记录
- 宁可标记"其他"也不要遗漏有价值的信息

## 查看模式

用户说"查看记录"、"有多少待办"时：

```bash
lark-cli base +record-list \
  --base-token <base_token> \
  --table-id <table_id> \
  --limit 200
```

按用户关心的维度（标签/状态/时间）汇总展示。

---

## 限制

- 单条记录模式每次只处理一条（录音模式可批量）
- 标签选项：待办、灵感、其他、AI、社群、内容创作、备孕、线下（可扩展）
- 不要修改已存在的记录内容，只更新"处理状态"
- 用户提供多条内容时，逐条处理（每条都需去重检查）
- 写入失败时报告失败原因
- 文章/链接类内容必须写入文章收藏表，不写入记录表
